📋 Project Overview Questions

1 Tell me about your project. What does it do?

FinTrack is a full-stack Personal Finance Tracker that helps users manage their income, expenses, and budgets. It features:

  • User Authentication — Secure registration and login using JWT tokens with BCrypt password hashing
  • Transaction Management — Full CRUD operations for income and expense tracking with category tagging
  • Budget Tracking — Users can set monthly spending limits per category and track their progress
  • Interactive Dashboard — Visual charts showing spending trends (line chart), category breakdown (doughnut chart), budget progress bars, and recent transactions

The backend is built with Java 17 + Spring Boot 3.3, the database uses H2 (dev) / MySQL (prod), and the frontend is a Single Page Application built with vanilla HTML/CSS/JS and Chart.js.

2 Why did you choose this tech stack?
  • Spring Boot — Industry standard for Java backend development. Provides auto-configuration, embedded server, and a vast ecosystem (Security, Data JPA, etc.)
  • H2 Database — Zero-configuration embedded database, perfect for development. Can easily switch to MySQL/PostgreSQL for production by changing one property
  • JWT Authentication — Stateless authentication ideal for REST APIs. No server-side sessions needed, scales horizontally
  • Vanilla JS Frontend — Demonstrates core JavaScript skills without framework abstraction. Shows understanding of DOM manipulation, Fetch API, and modular code organization
  • Chart.js — Lightweight, responsive charting library that integrates easily without heavy dependencies
3 What architecture/design pattern does your project follow?

The project follows a layered architecture (also known as N-tier architecture):

Client (Browser)
    ↓ HTTP/JSON
Controller Layer  → Handles HTTP requests, input validation
    ↓
Service Layer     → Business logic, orchestration
    ↓
Repository Layer  → Data access, database queries
    ↓
Database (H2/MySQL)

Design patterns used:

  • Repository Pattern — Spring Data JPA repositories abstract database operations
  • DTO Pattern — Data Transfer Objects decouple API responses from entity structure
  • Builder Pattern — Lombok's @Builder for clean object construction
  • Filter Chain Pattern — Spring Security's filter chain for request processing
  • Singleton Pattern — Spring beans are singletons by default

☕ Spring Boot & Backend Questions

4 What is Spring Boot and why use it over plain Spring?

Spring Boot is an opinionated framework built on top of the Spring Framework that simplifies configuration:

  • Auto-configuration — Automatically configures beans based on classpath dependencies. If you add spring-boot-starter-data-jpa, it auto-configures a DataSource, EntityManager, etc.
  • Embedded Server — Ships with Tomcat embedded, so you run mvn spring-boot:run instead of deploying a WAR to an external server
  • Starter POMs — Pre-configured dependency bundles. spring-boot-starter-web brings in everything needed for REST APIs
  • Production-ready — Built-in health checks, metrics, externalized configuration

In my project, I used starters for Web, Data JPA, Security, and Validation — each bringing in 10+ transitive dependencies that are pre-tested for compatibility.

5 Explain Dependency Injection. How did you use it?

Dependency Injection (DI) is a design principle where objects receive their dependencies from an external source rather than creating them internally.

// WITHOUT DI (tightly coupled)
public class AuthController {
    private AuthService service = new AuthService(); // Bad!
}

// WITH DI (loosely coupled) — What I used
@RestController
@RequiredArgsConstructor  // Lombok generates constructor
public class AuthController {
    private final AuthService authService; // Spring injects this
}

Benefits: Loose coupling, easier testing (can mock dependencies), follows SOLID principles (Dependency Inversion).

In my project, I used constructor injection via Lombok's @RequiredArgsConstructor across all controllers and services.

6 What are the different Spring annotations you used and what do they do?
  • @RestController — Marks a class as a REST API controller (@Controller + @ResponseBody)
  • @Service — Marks a business logic class as a Spring-managed service bean
  • @Repository — Marks a data access interface (auto-implemented by Spring Data JPA)
  • @Configuration — Declares a class that provides @Bean definitions
  • @Entity — Maps a Java class to a database table (JPA)
  • @GetMapping / @PostMapping / @PutMapping / @DeleteMapping — Maps HTTP methods to handler methods
  • @RequestBody — Deserializes JSON request body into a Java object
  • @PathVariable — Extracts a value from the URL path (e.g., /api/transactions/{id})
  • @RequestParam — Extracts query parameters (e.g., ?type=EXPENSE)
  • @AuthenticationPrincipal — Injects the currently authenticated user from the JWT token
  • @Valid — Triggers bean validation annotations (@NotBlank, @Email, etc.)
  • @Transactional — Wraps a method in a database transaction (rollback on failure)
7 What is a DTO and why did you use it instead of returning entities directly?

A Data Transfer Object (DTO) is a plain Java object used to transfer data between layers, especially between the API and the client.

Why not return entities directly?

  • Security — Entities may contain sensitive fields (e.g., passwordHash) that should never be sent to the client
  • Decoupling — API response shape shouldn't depend on database schema. If you rename a column, the API shouldn't break
  • Circular references — Entities with bidirectional relationships (User ↔ Transaction) cause infinite JSON recursion
  • Flexibility — DTOs can include computed fields (e.g., budgetPercentage) that don't exist in the entity

In my project, TransactionDTO includes categoryName, categoryIcon, categoryColor — data from the related Category entity flattened into the response for frontend convenience.

🗄️ Database & JPA Questions

8 What is JPA and Hibernate? How are they related?
  • JPA (Java Persistence API) — A specification (set of interfaces) that defines how Java objects map to database tables. It's like a contract.
  • Hibernate — An implementation of the JPA specification. It does the actual work of converting Java objects to SQL queries.

Analogy: JPA is like the List interface, and Hibernate is like the ArrayList implementation.

In my project, I defined entities with JPA annotations (@Entity, @Column, @ManyToOne), and Hibernate auto-generated the DDL (CREATE TABLE) and all SQL queries at runtime.

9 Explain the entity relationships in your project.
User (1) ──── (N) Transaction
User (1) ──── (N) Budget
Category (1) ── (N) Transaction
Category (1) ── (N) Budget
  • @ManyToOne — A Transaction belongs to ONE User and ONE Category. The foreign key (user_id, category_id) is on the Transaction table.
  • @OneToMany — A User has MANY Transactions. This is the inverse side (no foreign key here).
  • FetchType.LAZY — Related entities are loaded only when accessed, not eagerly with every query. This prevents loading thousands of transactions when you just need a user's name.
  • CascadeType.ALL — On User → Transactions: if you delete a user, all their transactions are deleted too.
10 What is the N+1 query problem? Does your project handle it?

The N+1 problem occurs when loading a list of entities triggers N additional queries for related data:

  • 1 query to load 100 transactions
  • 100 queries to load each transaction's category (one per transaction)

How I handle it:

  • Using DTOs instead of returning entities — the service layer maps entities to DTOs in Java, not in JSON serialization
  • Using custom JPQL queries with explicit joins for dashboard aggregations (e.g., GROUP BY t.category)
  • Using @ManyToOne with FetchType.EAGER on Transaction → Category (category is small, always needed)
11 What is the difference between ddl-auto=update vs create vs validate?
  • create — Drops all tables and recreates them on every startup. Destroys all data!
  • create-drop — Same as create, but also drops tables on shutdown
  • update — Adds new columns/tables but never deletes. Safe for development (what I used)
  • validate — Only checks if entities match the schema. Throws error if they don't. Safe for production
  • none — Does nothing. Use with manual migration tools like Flyway
💡 Production best practice: Use validate with a migration tool like Flyway or Liquibase for version-controlled schema changes.
12 Why did you use BigDecimal for money instead of double?

double uses floating-point representation which causes rounding errors:

double result = 0.1 + 0.2;  // 0.30000000000000004 ❌

BigDecimal uses exact decimal representation:

BigDecimal result = new BigDecimal("0.1")
    .add(new BigDecimal("0.2"));  // 0.3 ✅

For financial applications, even a tiny rounding error compounded over thousands of transactions can lead to incorrect balances. Always use BigDecimal for money.

🔐 Security & JWT Questions

13 How does JWT authentication work in your project?
1. REGISTRATION:
   Client → POST /api/auth/register {username, password}
   Server → Hash password with BCrypt → Save to DB
   Server → Generate JWT token → Return to client

2. LOGIN:
   Client → POST /api/auth/login {username, password}
   Server → Verify password against BCrypt hash
   Server → Generate JWT token → Return to client

3. AUTHENTICATED REQUESTS:
   Client → GET /api/transactions
            Header: Authorization: Bearer eyJhbGciOi...
   Server → JwtAuthFilter intercepts the request
          → Extracts token from Authorization header
          → Validates token (signature, expiration)
          → Loads user from DB
          → Sets SecurityContext
          → Request proceeds to controller

The JWT token contains the username in its payload (claims) and is signed with a secret key using HS512 algorithm. It expires after 24 hours.

14 What is the difference between authentication and authorization?
  • Authentication = "Who are you?" — Verifying the user's identity (login with username/password)
  • Authorization = "What can you access?" — Checking if the authenticated user has permission to access a resource

In my project:

  • Authentication: JwtAuthFilter validates the JWT token and identifies the user
  • Authorization: SecurityConfig defines that /api/auth/** is public and everything else requires authentication. Additionally, each service method checks that users can only access their own data (e.g., a user can't see another user's transactions)
15 Why did you disable CSRF? Isn't that a security risk?

CSRF (Cross-Site Request Forgery) is an attack where a malicious website tricks a user's browser into making unwanted requests to your server using their existing session cookies.

Why I disabled it:

  • CSRF attacks exploit cookie-based sessions
  • My API uses JWT tokens in the Authorization header, not cookies
  • JWT tokens are not automatically attached by the browser (unlike cookies), so CSRF attacks can't work
  • REST APIs are typically stateless and don't use sessions
💡 Rule: If you use cookies for auth → enable CSRF. If you use JWT in headers → CSRF protection is unnecessary.
16 Why BCrypt for password hashing? How is it different from SHA-256?
  • SHA-256 — Fast hash function. An attacker can try billions of password guesses per second.
  • BCrypt — Intentionally slow hash function with configurable "work factor". Takes ~100ms per hash, making brute-force attacks impractical.

BCrypt also includes a random salt in each hash, so the same password produces different hashes:

"password123" → $2a$10$dXJ3SW6G7P50lGmM...  (first time)
"password123" → $2a$10$xYz8Abc9Def1Ghi...  (second time, different!)

This prevents rainbow table attacks (precomputed hash lookups).

🌐 REST API & Frontend Questions

17 What are REST principles and how does your API follow them?
  • Resource-based URLs — /api/transactions (noun, not verb like /getTransactions)
  • HTTP Methods as verbs — GET (read), POST (create), PUT (update), DELETE (remove)
  • Stateless — Each request is self-contained with the JWT token. Server stores no session state
  • Proper HTTP Status Codes — 200 (OK), 201 (Created), 204 (No Content for DELETE), 400 (Bad Request), 401 (Unauthorized), 404 (Not Found)
  • JSON format — Request and response bodies use JSON
GET    /api/transactions      → List all (200)
GET    /api/transactions/42   → Get one (200 or 404)
POST   /api/transactions      → Create (201)
PUT    /api/transactions/42   → Update (200)
DELETE /api/transactions/42   → Delete (204)
18 What is CORS and how did you handle it?

CORS (Cross-Origin Resource Sharing) is a browser security mechanism. When the frontend (http://127.0.0.1:5500) calls the backend (http://localhost:8080), the browser blocks it because they're different origins.

How I handled it:

  • Created a CorsConfigurationSource bean that allows specific origins, methods, and headers
  • Integrated it into Spring Security's filter chain using .cors()
  • This is critical because Spring Security processes requests before Spring MVC, so CORS must be handled at the security level
💡 A common bug I fixed: Initially, CORS was configured in Spring MVC only (WebMvcConfigurer), but Spring Security blocked OPTIONS preflight requests with 403. The fix was adding .cors(cors -> cors.configurationSource(...)) in SecurityFilterChain.
19 How did you handle errors/exceptions globally?

I created a @RestControllerAdvice class called GlobalExceptionHandler that catches all exceptions and returns consistent JSON error responses:

{
  "status": 404,
  "error": "Not Found",
  "message": "Transaction not found with id: 42",
  "timestamp": "2026-06-01T10:30:00"
}

It handles:

  • ResourceNotFoundException → 404
  • MethodArgumentNotValidException (validation failures) → 400 with field-level errors
  • IllegalArgumentException → 400
  • Generic Exception → 500 (catch-all)

Without this, Spring returns raw stack traces or inconsistent error formats.

🧠 Problem-Solving & Behavioral Questions

20 What was the most challenging part of this project?

The most challenging part was integrating Spring Security with CORS and JWT. Specifically:

  • The CORS preflight OPTIONS requests were being blocked by Spring Security's filter chain with 403 Forbidden
  • This was tricky to debug because the browser only showed "Failed to fetch" — not the actual CORS error
  • I had to understand the request processing order: Spring Security filters run before Spring MVC's CORS handler
  • The fix was moving CORS configuration into a CorsConfigurationSource bean and integrating it directly into the SecurityFilterChain with .cors()

This taught me the importance of understanding the full request lifecycle in a Spring application.

23 What is @Transactional and why is it important?

@Transactional wraps a method in a database transaction. If any part of the method fails, ALL database changes are rolled back.

@Transactional
public void transferMoney(Long from, Long to, BigDecimal amount) {
    deductFromAccount(from, amount);   // Step 1
    addToAccount(to, amount);          // Step 2 (if this fails,
                                       //  Step 1 is rolled back!)
}

I used @Transactional on service methods that perform multiple database writes (e.g., creating a transaction and updating budget tracking). This ensures data consistency — you never end up in a half-completed state.

24 How do you ensure one user cannot access another user's data?

I implemented ownership verification at the service layer:

private Transaction findTransactionByIdAndUser(Long id, String username) {
    Transaction transaction = transactionRepository.findById(id)
        .orElseThrow(() -> new ResourceNotFoundException("Transaction", "id", id));

    // Security check: does this transaction belong to the requesting user?
    if (!transaction.getUser().getUsername().equals(username)) {
        throw new ResourceNotFoundException("Transaction", "id", id);
    }
    return transaction;
}

Key points:

  • The username comes from the JWT token (via @AuthenticationPrincipal), not from the request body — so it can't be faked
  • If a user tries to access someone else's transaction, they get a 404 Not Found (not 403) — this avoids leaking information about the existence of the resource
  • All CRUD operations go through this check
25 Explain the flow of a request from frontend to database and back.
1. User clicks "Add Transaction" → fills form → clicks Save

2. Frontend (transactions.js):
   fetch("http://localhost:8080/api/transactions", {
     method: "POST",
     headers: { "Authorization": "Bearer eyJ...", "Content-Type": "application/json" },
     body: JSON.stringify({ categoryId: 1, amount: 500, type: "EXPENSE", ... })
   })

3. Spring Security (JwtAuthFilter):
   → Extracts JWT from Authorization header
   → Validates signature and expiration
   → Loads user "testuser" from database
   → Sets SecurityContext → Request authorized ✅

4. Controller (TransactionController):
   @PostMapping → receives TransactionDTO + @AuthenticationPrincipal
   → Calls transactionService.createTransaction(dto, username)

5. Service (TransactionService):
   → Looks up User and Category entities from repositories
   → Creates Transaction entity with Builder pattern
   → Calls transactionRepository.save(transaction)

6. Repository → Hibernate → SQL:
   INSERT INTO transactions (user_id, category_id, amount, type, ...) VALUES (1, 1, 500, 'EXPENSE', ...)

7. Response flows back:
   Transaction entity → toDTO() → TransactionDTO → JSON → HTTP 201 → Frontend
   → Frontend shows toast "Transaction added!" and refreshes the dashboard
